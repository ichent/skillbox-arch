1. Масштабируемость (Scalability)
    1. Описание: Способность системы увеличивать производительность пропорционально добавляемым ресурсам для обработки растущей нагрузки.
    2. Ключевые аспекты:
        * Горизонтальное масштабирование (возможность добавления новых серверов).
        * Эластичность (автоматическое выделение ресурсов при пиковых нагрузках).
        * Балансировка нагрузки (равномерное распределение запросов).
    3. Например:
        * При старте массового онлайн-марафона (10 000 участников одновременно) серверная часть автоматически подключает дополнительные вычислительные мощности.
        * База данных успешно справляется с ростом количества сохраняемых тренировок с 1 миллиона до 10 миллионов без переписывания архитектуры.
2. Производительность (Performance)
    1. Описание: Характеристика системы, определяющая время отклика и пропускную способность при обработке запросов пользователей.
    2. Ключевые аспекты:
        * Время отклика API (Latensy).
        * Скорость обработки аналитических данных.
        * Оптимизация запросов к базе данных.
    3. Например:
        * Лента новостей и результаты друзей загружаются не медленнее 1.5 секунд.
        * Пересчет рейтинга пользователя после завершения тренировки занимает не более 5 секунд.
3. Безопасность (Security)
    1. Описание: Свойство системы обеспечивать конфиденциальность, целостность и доступность данных, защищая их от несанкционированного доступа.
    2. Ключевые аспекты:
        * Защита персональных данных (шифрование, соответствие GDPR/152-ФЗ).
        * Аутентификация и авторизация пользователей.
        * Защита от внешних атак (DDoS, XSS, SQL-инъекции).
    3. Например:
        * Данные о здоровье (пульс, маршруты тренировок) хранятся в зашифрованном виде.
        * Доступ к администрированию групп имеют только верифицированные организаторы.
4. Доступность (Availability)
    1. Описание: Способность системы оставаться работоспособной и доступной для пользователей в согласованное время.
    2. Ключевые аспекты:
        * Минимизация времени простоя (Downtime).
        * Резервирование критически важных узлов.
        * Мониторинг доступности сервисов.
    3. Например:
        * Сервис приема данных тренировок доступен 99.9% времени (не более 43 минут простоя в месяц).
        * При обновлении версии сервера пользователи не теряют доступ к основному функционалу (Zero-downtime deployment).
5. Надежность (Reliability)
    1. Описание: Способность приложения корректно выполнять свои функции без сбоев в течение определенного периода времени и в неблагоприятных условиях.
    2. Ключевые аспекты:
        * Устойчивость к отказам сети (Offline-first подход).
        * Гарантированная доставка сообщений.
        * Целостность данных при сбоях оборудования.
    3. Например:
        * При потере связи во время пробежки приложение продолжает записывать трек локально.
        * При сбое сервера в момент отправки результата, приложение автоматически повторяет попытку позже, не теряя данные.
6. Совместимость / Интероперабельность (Interoperability)
    1. Описание: Способность системы эффективно взаимодействовать с внешними компонентами, устройствами и системами.
    2. Ключевые аспекты:
        * Поддержка стандартизированных протоколов передачи данных.
        * Интеграция со сторонними API (социальные сети, фитнес-платформы).
        * Корректная работа с разнородным оборудованием (IoT датчики).
    3. Например:
        * Приложение корректно считывает данные пульса как с Apple Watch, так и с нагрудного датчика Garmin.
        * Возможность экспорта тренировки в формате .GPX для использования в других сервисах.
7. Модифицируемость (Modifiability)
    1. Описание: Лёгкость, с которой в систему могут быть внесены изменения (добавление новых функций, изменение бизнес-логики) без нарушения работы существующего функционала.
    2. Ключевые аспекты:
        * Слабая связность компонентов.
        * Подвижность бизнес-правил (возможность менять параметры геймификации без перекомпиляции).
        * Модульная структура кода.
    3. Например:
        * Маркетолог может изменить правила начисления баллов за «Утреннюю пробежку» через админ-панель, не привлекая разработчиков.
        * Добавление нового вида спорта (например, «Сап-серфинг») в справочник инвентаря занимает минимальное время.
